## 1. Обзор

**Сеттинг:** Милан, конец 1470-х (после смерти Галеаццо Мария Сфорца в 1476 — вокруг малолетнего Джан Галеаццо Сфорца и борьбы за регентство). Не военная осада (как в двух существующих сценариях), а политическая борьба за легитимность при слабом центре — военная сила вторична по отношению к союзам, бракам и деньгам.

**Условие победы:** объединение Италии.

**Продолжение после победы — уже поддерживается движком, ничего нового строить не нужно.** `GameMode::Scenario → Consequences → Free` в `core/world.rs`. Переход Scenario→Consequences — автоматический через `MilestoneEvent` с `triggers_collapse: true` (название условное, механизм универсален). Consequences→Free — ручной переход к свободной игре (`application/modes.rs`). Milestone "Италия объединена" играет ту же роль, что обычно играет коллапс-milestone.

**Главные акторы:** Милан (игрок), Венеция, Флоренция, Неаполь, Папская область, Франция (на подходе, не гарантированный участник событий).

**Второй тир:** Генуя, Феррара (Эсте), Мантуя (Гонзага), Сиена, Урбино (Монтефельтро), Болонья (Бентивольо), Монферрат/Савойя.

**Юг:** Сицилия — отдельный полюс силы, не сателлит Неаполя (арагонская корона напрямую, другая ветвь той же династии, что и Ферранте I Неаполитанский). Баронская фронда и османская угроза (Отранто, 1480) — не отдельные акторы, а внутренняя нестабильность/растущее давление на Неаполь конкретно.

**Не взято:** Модена (личная уния с Феррарой на этот момент, разделение не даёт самостоятельной ценности), внутримиланские фракции сверх герцога (усложняет без необходимости), Таранто (уже поглощён короной к 1463).

## 3. Находки в движке — контекст для всех заданий ниже

- **`GameMode::Consequences/Free`, `calculate_trade/diplomatic/military_interaction`** — уже реализованы, переиспользуются как есть.
- **`Alliance` struct в `core/world.rs` — мертва**, ни одна функция её не создаёт/не читает. Не трогать в этом сценарии (v2).
- **`InteractionType::Vassalage` объявлен, но не реализован** — расчётной функции нет. Задание A.
- **Нет механизма завоевания/аннексии** — единственный способ исчезновения актора — коллапс (`check_collapses` в `engine/mod.rs`, жёстко закодированные пороги `legitimacy<10, cohesion<15, external_pressure>85`) + наследование по `successors`.
- **`external_pressure` — скаляр без атрибуции источника.** Сюзерен для вассалитета вычисляется на лету в момент триггера, не хранится постоянно.
- **`MilestoneEvent`-эффекты захардкожены по ID** (`apply_milestone_effects`, единственный пример — `mehmed_accelerates`). Не годится для повторяемых событий.
- **`RandomEvent` — уже полностью data-driven**: `{ id, probability, target: EventTarget, conditions, effects: HashMap<String,f64>, llm_context, one_time }`. Используется для меценатства и коалиции (задания C, D) — ноль правок движка.
- **`EventCondition`/`Condition` не умеют считать акторов** — только `Metric` (один актор/глобально), `ActorState`, `Tick`. Решение для коалиции — счётчик-метрика (см. задание D).
- **`spawn_actor` (используется через `MilestoneEvent`, движковый уровень, `check_milestone_events` в `engine/mod.rs`) создаёт нового актора без соседей** — хардкодит `neighbors: vec![]`, не даёт способа задать список соседей появляющемуся актору. Все взаимодействия (`calculate_military/trade/diplomatic/migration_interaction`) работают строго по парам из `get_neighbor_pairs`, построенным из декларированных `neighbors` — актор без них физически не может участвовать ни в одном взаимодействии после появления. Найдено эмпирически на `france_intervenes` в задании E (см. ниже): Франция появляется на сцене (событие `spawn_france`, актор реально существует в `world.actors`), но механически инертна — просто присутствует, ничего не делает и ни на что не влияет. **Тот же класс проблемы, что уже трижды всплывал в этом документе** — `Alliance` (мертва), `InteractionType::Vassalage` (было нереализовано до задания A), захардкоженные `MilestoneEvent`-эффекты (`apply_milestone_effects`, единственный пример `mehmed_accelerates`): существует, выглядит рабочим, декоративно по факту. Касается любого будущего сценария, который попробует содержательно использовать `spawn_actor` (не только Франция здесь) — новый появившийся актор нужно либо снабжать соседями вручную постфактум (нет API для этого), либо чинить на уровне движка. Код не тронут, только зафиксировано.

## Задание A — Вассалитет (новая механика движка)

**Модель: Opus.** `/clear` перед стартом, прочитать `AGENTS.md`. Отдельная ветка, отдельный PR, обязательная baseline-верификация через `cargo run --bin sim` до/после **на обоих существующих сценариях** (rome_375, constantinople_1430) — новая логика не должна их задеть, хотя они вассалитет не используют.

**Иерархия запрещена** — у вассала не может быть своего вассала. Проверка при формировании: кандидат не должен уже быть чьим-то сюзереном.

### Данные
```rust
struct Vassalage {
    vassal_id: String,
    overlord_id: String,
    formed_tick: u32,
}
```
`Vec<Vassalage>` на `WorldState` (по образцу мёртвой `Alliance`, но рабочая).

### Триггер — `check_vassalage`, параллельно `check_collapses`
Полоса между "давление серьёзное" и "гибнет":
- `external_pressure`: 70–85
- `legitimacy`: 10–25
- `cohesion`: 15–30

3 тика подряд в полосе (отдельный счётчик по аналогии с `collapse_warning_ticks`).

**Атрибуция сюзерена:** на лету в момент триггера — кто из соседей даёт наибольшее военное давление (по данным `get_neighbor_pairs`), не хранить историю.

### Эффект — достроить `calculate_vassalage_interaction`
Дань раз в тик: **3–5% `economic_output` вассала**, симметрично из `treasury` вассала в `treasury` сюзерена. Через `treasury`, не `economic_output` напрямую (архитектурный прецедент торговли).

**Намеренно вырезано:** подавление роста метрик вассала — отдельный тестируемый рычаг, не часть этого PR.

### Выход / восстание
По паттерну `cultural_power` (`a_power > b_power * 1.5`): распадается, если военная мощь вассала ≥ 80% мощи сюзерена, **или** сюзерен сам пересекает полосу 70–85. Отдельное событие в `EventLog`.

### Счётчик для будущей коалиции (задание D)
Побочный эффект в этой же функции: `overlord.add_metric("expansion_count", 1.0)` при формировании вассалитета **и** при полном поглощении через коллапс+наследника (оба пути — один общий счётчик, не раздельные).

## Отложено в v2 — и почему

| Пункт | Причина |
|---|---|
| **Альянс** (устойчивый, N≥2, меняет взаимные расчёты) | Та же сложность, что Вассалитет. Делать оба сразу в одном сценарии — два независимых новых блока логики движка перед первым релизом, сложнее диагностировать проблему при плейтесте. |
| **Подавление роста метрик вассала** | Отдельный тестируемый рычаг, не обязателен для работы самой идеи вассалитета. |
| **Обобщённая реактивная система коалиций / противовесов** (не скриптованная через milestone/RandomEvent, а системная реакция на любой дисбаланс силы между любыми акторами) | Самый ценный, но и самый дорогой пункт. Проектировать как отдельный движковый проект, когда появится второй сценарий, которому это тоже нужно — не вслепую под один кейс. **Больше не только гипотеза** — см. находку про `affinity`/Savoy в разделе "Найдено при плейтесте C/D" ниже: конкретное эмпирическое доказательство, что без системы противовесов военная динамика в культурно-однородном сценарии вырождается в детерминированный исход, а не просто теоретический риск на будущее. |

## Найдено при плейтесте C/D — отслеживаемый вопрос, не решён

**`vassalage-pressure-reachability`.** При плейтесте заданий C/D (15 сидов × 300 тиков) вассалитет (задание A) ни разу не сформировался. Причина — не баг Milan 1477: **у `external_pressure` нет пути распада нигде в движке, во всех трёх сценариях** — метрика только растёт (военные конфликты, auto_deltas) до насыщения в 100. Полоса вассалитета (70–85) — переходное окно во время монотонного роста к насыщению, а не стабильная достижимая зона. Комбинируется с уже существующим общим правилом `external_pressure_legitimacy_to_cohesion_bonus` (то же в rome_375), которое поднимает `cohesion` при пересечении 65 — из-за чего `cohesion` у насыщенного актора запирается около 99, постоянно вне окна вассалитета (15–30).

**Это может быть пробелом в самой верификации задания A**, не в Milan 1477: тесты A проверяли поведение на синтетических акторах с вручную выставленными метриками прямо внутри полосы — не через реальную tick-by-tick динамику. Доказано "если актор попадёт в полосу — сработает правильно", но никогда не доказано "актор реально может попасть в полосу при живой игре".

**Попутно исправлен реальный, подтверждённый баг** (не то же самое, что находка выше): `tags.toml` для Milan 1477 давал 7 тегам заразный (contagious) спред `legitimacy`/`cohesion`, чего нет ни в одном из двух существующих сценариев. Исправлено, поведение проверено как настоящее (метрики теперь колеблются, а не монотонно ползут к 100 и там остаются) — не просто отложенное насыщение.

**Что решать отдельно, по образцу D3 (сначала подтвердить гипотезу экспериментом на всех трёх сценариях, потом решать про правку):** нужен ли `external_pressure` механизм распада на уровне движка, и если да — какой. Не чинить точечной настройкой чисел в конкретном сценарии — это замаскирует архитектурный вопрос под сценарийный хак.

**Проверено плейтестом (те же 15 сидов × 300 тиков): коллапс НЕ даёт запасного пути для `expansion_count`.** Коллапс действительно происходит — ровно 1 раз в каждом из 15 сидов, всегда один и тот же актор (`savoy`, тик ~42–53), что согласуется с насыщением `external_pressure` выше порога 85. Но `expansion_count` не растёт и через этот путь: у всех 13 акторов Milan 1477 `on_collapse: vec![]` (задание B сознательно не завело `successors` — см. заметку задания B про `is_successor_template`). Ветка `check_collapses`, начисляющая `expansion_count` наследнику (`if let Some(heir) = world.actors.get_mut(&successor.id)`), физически не выполняется, если у актора нет ни одного successor'а — она просто не достигается.

**Вывод: оба пути к `expansion_count` сейчас мертвы одновременно, по независимым причинам** — вассалитет упирается в архитектурный вопрос про `external_pressure` (см. выше), поглощение через коллапс упирается в то, что задание B не сконфигурировало `on_collapse` ни для одного актора. Это два разных фикса, не один: чтобы коллапс-путь заработал, заданию B/E нужно добавить `successors` хотя бы части акторов (например, слабый сосед → сюзерен/сильный сосед как наследник) — это чисто контентная правка, не требует трогать движок и не зависит от решения по `external_pressure`. Задание E должно либо (а) дождаться решения по вассалитету и добавить successors для коллапс-пути, либо (б) спроектировать условие победы вокруг метрики, не зависящей от `expansion_count` вовсе.

**Третья, отдельная находка — почему коллапсится именно `savoy` (детерминированно, не по случайности сида): у Milan 1477 всё население — один culture/religion, force projection в движке из-за этого полностью отключён.** Проверено `EFFMIL`-трейсом внутри `calculate_military_interaction`: `effective_military` у **любого** актора в этом сценарии тождественно равен его сырому `military_size`, независимо от числа соседей. Причина в формуле `affinity()`: `cultural_affinity(same culture) = 0.2`, `religious_modifier(same religion) = -0.2`, сумма = `0.0` (clamp [0,1] не спасает — уже ровно на нижней границе). Все 13 акторов Milan 1477 — `Culture::Latin` + `Religion::Catholic` без исключений (это исторически корректно — все они итальянские католические города, но как побочный эффект `divisor = neighbor_count × avg_affinity` всегда получает `0.0`, упирается в `.max(1.0)`, и `effective_military = raw / 1.0 = raw` для каждого актора, каждый тик). Сравнение с существующими сценариями подтверждает: rome_375 использует 8 разных культур/5 религий, constantinople_1430 — 5 культур/3 религии — там force projection реально работает.

Итог для `savoy` конкретно: `military_size` стартует на 20 против 40 у соседа Милана (сосед №2 — Генуя, mil=18, слабее). Без force-projection нюанса (который в других сценариях мог бы скорректировать разрыв через топологию соседей) исход конфликта Милан-vs-Savoy решается чистым сравнением `raw military_size` — Милан почти всегда атакующий. Поскольку `defender_loss` (15–30%) systematически больше `attacker_loss` (5–15%), а нигде в движке нет механизма, который бы **растил** `military_size` обратно (только штрафующие deficit/excess-правила), разрыв только увеличивается с каждым конфликтом — гарантированный снос к нулю, не вероятностный исход. Именно поэтому во всех 15 сидах коллапсится ровно `savoy` и никто другой (не Сиена, хотя у неё стартовые legitimacy/cohesion даже ниже) — величина случайна (RNG решает точный тик через `interaction_cooldowns`/roll), а сам факт — нет.

**Это не баг движка** (формула `affinity()` работает штатно для сценариев с реальным культурным разнообразием) **и не баг Milan 1477 per se** (историческая точность — единая католико-латинская Италия оправдана), но это значимое для баланса B/D/E побочное следствие: контент-автор, работающий над Milan 1477 (в т.ч. задание E), должен знать, что force projection здесь ничего не делает и весь военный баланс сводится к сырым `military_size`/`military_quality` соседей. Не чинить точечно без явного решения — как минимум задокументировать, при желании обсудить отдельно (не входит в объём C/D).

**Эта находка — не изолированный факт про Milan 1477, а первое реальное эмпирическое обоснование v2-пункта "Обобщённая реактивная система коалиций / противовесов"** (см. таблицу "Отложено в v2" выше). До этого плейтеста тот пункт был гипотетическим риском на будущее ("может пригодиться другому сценарию"); теперь есть конкретный, воспроизводимый на 15/15 сидах пример того, как отсутствие системы противовесов (никто не вмешивается, когда один сосед лавинообразно давит другого) вырождает военную динамику в предопределённый исход в культурно-однородном сценарии. Довод в пользу того, что система противовесов из v2 нужна не абстрактно "когда появится второй сценарий", а уже для полноценной живой партии в самом Milan 1477 — но по объёму работы это остаётся v2, не входит в C/D/E.

**Проверено под реальными действиями игрока, не только в вакууме.** Все находки про `military_size` Милана (и попытка найти для задания D сигнал, растущий вместе с реальным давлением Милана на соседей — `expansion_count(milan)`, `external_pressure(savoy)`, `military_size(milan)`) были получены из `run_single`/`batch`, где за всю партию не совершается ни одного игрового действия. Это ставило под сомнение сам вывод: возможно, тестировался безыгровой мир, а не сломанная механика роста.

Проверка показала — **milan_1477 вообще не имел scripted-стратегии** (`sim.rs`, `ScriptedStrategy::from_str` ветвился только на `rome_375` vs. всё остальное, где "всё остальное" хардкодило ID действий Константинополя — `venice_diplomacy`, `milan_bankers` и т.п., которых в `milan_1477/actions.toml` не существует). Эмпирически подтверждено: `cargo run --bin sim milan_1477 N scripted balanced` до фикса — `Total actions applied: 0`, `Total actions rejected: 320`, каждое действие отклонено, потому что ID не совпадают. Это отдельная, самостоятельная проблема (сценарий нельзя было честно нагрузочно протестировать с игровым поведением) — исправлена здесь: добавлена `ScriptedStrategy::MilanAggressive` (реальные ID из `milan_1477/actions.toml`, приоритет — `milan_pressure_genoa`, `incite_baronial_revolt`, `milan_hire_condottieri`, и т.д.), плюс вывод статистики специфичный для Милана вместо константинопольского (`fed`/`byzantium`, бессмысленного для этого сценария).

**Первый прогон с реальными агрессивными действиями (`cargo run --bin sim milan_1477 60 scripted aggressive`): `military_size` Милана всё равно монотонно падал к `0.00` к тику ~43, на том же расписании, что и без единого действия.** 180 действий реально применено (казна ушла в −3779, `legitimacy` реально выросла до ~97) — игрок в этом прогоне реально и агрессивно действовал, не наблюдал. Но причина оказалась дешевле, чем "движок не умеет": **прежде чем принимать вывод о v2, проверено, есть ли в `rome_375`/`constantinople_1430` рабочий прецедент действия, растящего `military_size`.** Оказалось — есть, в обоих: `constantinople_1430/actions.toml` даёт `byzantium.military_size` несколько действий (+5.0, +4.0, +6.0 на базовые 8.0), `rome_375/actions.toml` даёт `rome.military_size +10.0` через `fund_defense`. Milan 1477 просто не скопировал эквивалентное действие — **тот же класс пробела, что уже дважды находили в `tags.toml` и `milestone_events.toml`: не докопировали структуру, а не "движок не может"**.

**Исправлено** — добавлено `milan_raise_troops` ("Сбор ополчения") в `milan_1477/actions.toml`: `milan.military_size += 8.0`, cost `treasury -45.0`, gate `treasury > 70.0` (по образцу существующих `milan_hire_*` действий). Добавлено в приоритет `ScriptedStrategy::MilanAggressive`. Переигран тот же скриптованный прогон: **`military_size` реально вырос выше стартового значения — пик 55.17 (от стартовых 40.0) к тику ~7–10**, `milan_raise_troops` применено 3 раза, прежде чем сработало. Рост подтверждён на честных данных — вывод "рычага роста не существует" был неверным, исправлен.

**Но это не значит, что задание D можно спокойно калибровать по `military_size` прямо сейчас.** В том же прогоне рост не удержался: наивная скриптованная стратегия тратит казну на всё сразу (`milan_banking_deal_florence`/`milan_court_patronage`/`milan_diplomacy_ferrara` каждый тик, без учёта устойчивости бюджета), казна уходит в минус за ~7 тиков, `milan_raise_troops` (gate `treasury>70`) становится недоступен, и та же самая механика "конфликты стачивают `military_size` без пути восстановления" в итоге всё равно доводит его до `0.00` — просто на ~9 тиков позже (тик 52 вместо 43), чем без действия вовсе. Это означает: рост `military_size` **зависит от дисциплины стратегии** (в отличие от прежней картины, где ничего не зависело от игрока вообще) — правдоподобный сигнал для D, но его калибровка требует более аккуратного тестирования.

**Прежде чем калибровать — проверено, не является ли дефицит бюджета Милана на старте (income < upkeep уже на тике 0) багом задания B.** Формула `apply_treasury`: доход = `economic_output × population × 0.001`, расход = `military_size × 0.8`. У Милана на старте: доход 19.5, расход 32.0 — дефицит −12.5/тик уже в момент начала партии, без учёта конфликтов и трат на действия. Проверка на всех 13 акторах Milan 1477 **и** на всех акторах `rome_375`/`constantinople_1430` показала: **старт в дефиците — универсальная норма во всех трёх сценариях, не баг**. В `rome_375` профицит на старте есть только у крупнейших держав (Рим, Сасаниды, Гупты, Восточная Цзинь); все второстепенные акторы (гунны, вестготы, остготы, вандалы и т.д.) стартуют в дефиците. В `constantinople_1430` профицит есть только у Османов (и Венгрии, net=0) — сама Византия, Венеция, Генуя, Милан, Папство стартуют в дефиците. Milan 1477 (дефицит у всех 13 акторов, включая самого Милана) полностью соответствует этому паттерну. Стартовые числа Милана не тронуты.

**Добавлена дисциплинированная scripted-стратегия** (`sim.rs`, ветка `scenario_id == "milan_1477"` в `run_scripted`): `milan_raise_troops` получает приоритет над всеми остальными действиями и применяется, только если `treasury > 70` (порог самого действия); все прочие действия (`patronage`/`legitimacy`/давление на соседей) финансируются **только из профицита** сверх этого порога — резерв в 70 никогда не расходуется на них. Также добавлена поддержка сида в `run_scripted` (5-й CLI-аргумент, раньше был захардкожен на `42`) — нужно было для честного повторного прогона на тех же 15 сидах.

**Результат на 15 сидах × 100 тиков (`cargo run --bin sim milan_1477 100 scripted aggressive <seed>`):** пик `military_size` — ровно `64.00` в каждом из 15/15 сидов (детерминированно: первые 3 тика движок не считает военные конфликты — "стабилизационный период" — так что первый рывок действий полностью свободен от RNG). Держится ≥40 в течение 8–18 тиков (варьируется по сидам — здесь уже влияет RNG боевых столкновений), затем падает и колеблется в диапазоне пониже до конца партии. Ни в одном из 15 сидов без единого игрового действия (`run_single`/`batch`, все предыдущие прогоны в этом документе) `military_size` не поднимался выше стартовых 40 — рост **действительно зависит от того, инвестирует ли игрок в войска**, а не происходит сам по себе.

**Утверждённый порог D: `military_size(milan) >= 50`.** Пересобрано, порог обновлён в `create_random_events()` (`italian_league_against_milan`). Финальный плейтест (те же 15 сидов, дисциплинированная стратегия): **`italian_league_against_milan` стреляет на тике 1 в 15/15 сидов** (внутри детерминированного RNG-свободного окна разгона), **`italy_unified` — на тике 39–67** (зависит от сида, коллапс Savoy по-прежнему управляется RNG боевых столкновений). Разрыв между D и E теперь реальный и большой (38–66 тиков) — коалиция реально успевает существовать как препятствие до победы, а не срабатывать в тот же тик, что и она.

**Явная оговорка по ограничению калибровки:** порог откалиброван **по единственной синтетической scripted-стратегии** (`MilanAggressive`, дисциплинированная версия — резерв `treasury>70` для `milan_raise_troops`, остальное только из профицита). Не проверено на смешанных стилях игры (дипломатия/экономика/меценатство вместо чистой военной экспансии) — реальный игрок в GUI может показывать другую динамику `military_size`, и порог 50 может не подходить одинаково хорошо для всех стилей игры. Это ограничение самой калибровки, а не доказанная универсальность числа 50 — если после релиза выяснится, что порог не подходит для распространённого стиля игры (например, слишком легко/слишком редко достигается), это законный повод для повторной калибровки, не признак ошибки в проделанной здесь работе.

**Четвёртая находка, обнаружена при подготовке задания E — все 6 milestone Milan 1477 были мертвы, включая `france_intervenes`.** `EventConditionType::Metric` в движке ожидает **голое** имя метрики (`"cohesion"`, `"external_pressure"`) плюс отдельное поле `actor_id: Option<String>` (см. `check_event_condition` — если `actor_id` равен `None`, условие безусловно `false`). `milestone_events.toml` использовал формат `metric = "actor:naples.external_pressure"` (склеенная строка без отдельного `actor_id`) — синтаксис верный для `RandomEvent::Condition`/`MetricRef`, но не для `MilestoneEvent`. Поле `actor_id` при отсутствии в TOML молча дефолтится в `None` (serde делает это автоматически для `Option<T>`, без варнинга), поэтому валидатор (`validate_metric_ref`, вызывается из `registry.rs`) тоже не ловил проблему — для голых имён метрик (`MetricRef::Global`) он вообще не проверяет `actor_id`. Подтверждено эмпирически: до фикса `world.milestone_events_fired` был пуст после 300 тиков в 15/15 сидах; rome_375 (акторы строятся напрямую в `.rs`, без TOML, с верным форматом — образец в `rome_375.rs`) в тех же условиях штатно стрелял `adrianople`.

**Исправлено** (все 6 записей: `metric` — голое имя, `actor_id` — отдельное поле, по образцу rome_375). После фикса — переигран тот же набор 15 сидов × 300 тиков:
- 5 из 6 milestone стреляют штатно в каждом из 15/15 сидов (`otranto_threat_rises`, `otranto_falls`, `baronial_fronde_erupts`, `milan_regency_crisis_deepens`; `milan_regency_stabilizes` не стреляет ни разу — legitimacy Милана никогда не держится ≥65 пять тиков подряд, что нарративно согласуется с сюжетом кризиса, а не стабилизации).
- **`france_intervenes` теперь стреляет в 15/15 сидов** (тик ~5–10, задолго до коллапса Savoy на тике ~42–53) — Франция теперь появляется в каждой партии, а не "непредсказуемым третьим игроком", как задумано в §1.
- **Но это не меняет калибровку.** Появление Франции никак не повлияло на динамику Savoy: `expansion_count` по-прежнему стабильно равен `1` во всех 15/15 сидах (те же коллапсы Savoy→Milan, тот же единственный надёжный путь). Причина — `spawn_actor` создаёт Францию без соседей, поэтому она механически инертна с момента появления; это движковая находка сама по себе, вынесена в §3 (не деталь отчёта по Франции).

## Порядок выполнения

**A (Opus, `/clear`) → B (Sonnet, `/clear`) → C и D параллельно (Sonnet, D зависит от счётчика из A, C зависит только от акторов из B) → E (Sonnet, зависит от B).**

Между A и B — обязательный `/clear`, разные модели. Между B и C/D — `/clear` не обязателен (обе Sonnet, но если сессия уже длинная — лучше почистить контекст перед новым блоком контента). C и D можно делать в любом порядке относительно друг друга.
